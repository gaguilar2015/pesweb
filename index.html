PES Matching at the SIB

Author

Gian Aguilar

PES Matching Process

This document explains the process used to perform household-level and individual-level matching for the 2023 Post-Enumeration Survey, which took place in the months of August-September 2023 as a quality assurance and coverage estimate exercise for the Belize 2022 Population and Housing Census.

To perform the matchings, the general guidelines set out by PES consultant Armando Levinson were followed. However, the implementation for the matching was original. This is one of the first automated systems for PES matching that are known, and the rest of the document will summarize the methods and algorithm.

The original guidelines called for a resource-intensive matching process, in which teams of matchers, reviewers and matching-supervisors are all involved in both household and individual level-matching. Nevertheless, in light of limited resources, the SIB team attempted to abstract, program and automate as much of the process as possible. There were no manual matchers, reviewers, or match supervisors - although there was a manual review and verification mechanism automated through a real-time web dashboard.

The resulting matching program and dashboard were immensely helpful and extremely promising. Although not perfect (there is certainly always the need for a manual inspection and verification step), the SIB team believes considers the automated matching a success and one that be further improved and refined to become an invaluable tool for PES exercises in the future.

Key Features and Automation

The following are some key features of the PES matching process.

1. Software and automation

The PES matching process was implemented in its entirety in the R software. R is a free, open-source programming language for statistical analysis, data processing and visualization.

For automation purposes, we used orchestration software to schedule, manage and run the R scripts that did the following things (among many others):

  1. Download the latest PES data from Survey Solutions.
  2. Run the entire matching process on the latest data.
  3. Save the results in a central database.

We had to ensure that the R scripts were executed continuously and in the correct order, at the correct time, and report any errors or failures.

2. Matching Algorithms

The matching algorithms were based on the general guidelines set out by Mr. Levinson. However, the algorithms expanded on the guidelines in order to account for the automated process that replaced all traditional human-based roles and activities.

There exist algorithms for matching households and individuals. Some algorithms use simple logical rules, some follow record-linkage procedures, and some incorporate machine-learning models.

All developed algorithms are original.

3. PES Dashboard

In addition to the R scripts developed to perform the matching process, a web-based, real-time PES Dashboard was also developed for the survey managers. Among many other things, this dashboard allowed for the progressive monitoring of the matching process, as well as other metrics and indicators.

The dashboard incorporated a visual review features. For example, all household-level matches could be examined and verified. There was an easy mechanism for “unlinking” a false match, thus cleaning and verifying the final matched dataset.

4. Reconciliation

The process for preparing the reconciliation visits was also automated. This document will not expand on this topic, but there was a process for automatically preparing all of the reconciliation forms that interviewers took into the field for the reconciliation visits. This was also an original, automated implementation.

Household Matching

Household-level matching guidelines

We first review the household-matching guidelines in the Matching Manual:

  1. Select an EA and identify both Census and PES household questionnaires.

  2. Perform two-way matching of households based on the following information:

    1. Geography: District, ED, Block, and Building numbers match.

    2. Name of the household head: Name matches, allowingfor minor spelling mistakes that do not change the pronunciation.

    3. Name of household members: Name of at least 1 other household member matches, allowing for minor spelling mistakes that do not change the pronunciation.

  3. The previous steps may be relaxed if no immediate match is found.

    • For example, the building number or even block number do not need to match if all other household items (name of head and members match).

    • Similarly, the head might vary between PES and Census, so matching must account for this as well.

    • If a PES-Census household match cannot be found within a EA, the search can extend to the surrounding EAs or even nationally. 5124

Household matching - Stage 1

The first level of household matching was also known as Stage-1 Matching, or Direct Matching. The broad goal in this step was to quickly identify the easy matches: household-pairs in the same physical location (ED, block, buiding) that appeared to be the same household after comparing the names in the corresponding household listings. This step was important because we want to identify and exclude as much matches as possible in this “easy” step before moving on to more time-consuming, computationally expensive processes and algorithms.

Preprocessing

To ready the datasets for name-based matching, we first do some cleanup and preparation. In order to be able to more accurately compare the listed household members in the PES and Census households, the following preparation was done in each dataset:

  1. The columns containing the household member names were rearranged and sorted based on age and sex.
    • This sorting would, in most cases, ensure that persons in the household would be in the same order in both the PES and Census household listings.
    • This would be useful during matching.
    • In the case of the PES, the names of out-movers were also included in this process and the in-movers and born-afters were removed.

  1. A last_name variable was derived for each household to hold the last name of the listed head. This would also be useful during matching.

  2. An id variable, hh_id, was derived to uniquely identify each household.

    • Importantly, this ID was a concatenation of the ED number, building number, dwelling number and household number. Knowing this is useful during visual inspection.
  1. The name columns were cleaned up so that invalid values (“Missing”, “NA”, “None”) could be replaced with empty strings (’’).

Matching Algorithm

As previously explained, this first level of matching was a strict, full matching where ALL variables had to match. The following sections provide an expanded description of the full, direct-matching (Stage-1) algorithm.

1. Merging

Both datasets were merged, or joined, using the unique hh_id. This means that PES records were joined with Census records if they shared the same ED, building, dwelling and household numbers. In other words, records that were in the same exact geographic location were joined.

  2. Because both datasets share columns with the same name (“PersonN”), the merging algorithm assigned the suffix .x to the PES columns and .y to the Census columns, wherever there was a conflict.

  3. We can already see some characterstics of the data structure, in which records that appear to be for the same households have many spelling inconsistencies in the member names.

  4. We can also see the existence of records that likely do not belong to the same household (abisai alas).

This leads us to the next step.

2. Scoring pairs

As noted in the previous step, simply matching households based on geographic location is not enough. The final decision must be made on the basis of the listed household members. How do we compare the member names in an automated way? We cannot use an exact-match method, as there will always be inconsistencies in spelling, even when the names belong to the same person. Instead, we need a more flexible algorithm.

In the world of string matching/string search, there are many algorithms that are useful for evaluationg the similarity of two pieces of text (otherwise known as strings in the programming world). For the purpose of this exercise, I selected the Jaro-Winkler algorithm.

Note

The Jaro-Winkler distance algorithm is a measure of the similarity between two strings. It is a variant of the Jaro similarity algorithm, which compares the two strings character by character and takes into account the number of matching characters and the number of transpositions needed to transform one string into the other. JW gives more weight to similarities/differences at the start of the strings.

Now, we compare the similarity of the names of the household members using the JW algorithm. This was done in the following way:

  1. The name of Person1 in the PES dataset was compared with the name of Person1 in the Census dataset, PES Person2 with Census Person2, and so on for the up to 5 household members (add footnote).
    • Remember that we have previously sorted names in both datasets by age and gender, which will maximize chances that we are comparing the correct persons regardless of the order of the original PES and Census member listings.
  1. A similarity score was obtained and saved for each comparison.
  2. The maximum score for each record was also saved in a separate variable.
    • For example, if for a specific record, the Person2 comparison received the highest score, perhaps a perfect 1 if the name was exactly in both datasets, this score was saved in its own variable.

Now, we have a useful set of scores based on the similarity of the member names in the PES and Census households. How do we decide which are successful matches? In other words, how do we know if the two records are for the same household?

3. Selecting matches

The simplest way of selecting a match (and the one I chose) is to establish a treshold for the similarity scores and selecting those records that pass the treshold.

After some experimenting and visual inspection, I selected a similarity score of 0.7 as the matching treshold. Essentially, this means that if any of the name comparisons obtained a Jaro-Wrinkler score greater than 0.7, the household pair was determined to be a successful match.

Note

The 0.7 treshold is somewhat lenient, but given that in this Stage-1 Matching, we have already proven that the pair of households are in the exact same geographic location, then we should only need moderate proof that the household member names are similar in order to call it a match with reasonable confidence. This, of course, is an arbitrary choice that may be debated and modified.

In addition, each record that was selected as a match was assigned a confidence score, depending on how exactly the member names matched. The confidence score was somewhat complex, taking into consideration several factors (head last names, number of matching names, etc.), but a simplified version of assinging a score can be seen below:

  1. High - (Max JW score > 0.95)
  2. Moderate - (Max JW score > 0.85)
  3. Fair - (Max JW score >= 0.7)

Household Matching - Stage 2

It is worth noting that the Stage-1 matching would only find successful household pairs if both Census and PES households were recorded to be in the exact same physical/geographical location. However, variations in canvassing and interviewer error could mean that a match was missed due to enumerator error. For example: although in reality a household X is in the same geographic location for both PES and Census, the enumerator made an error when recording the building or dwelling number.

In Stage-2 matching, we will work with all remaining unmatched households within each ED. At this level, every remaining PES household in the ED will be compared against every remaining Census household in the ED.

To find matches, a special record-linkage procedure was used, in conjuction with a machine learning algorithm that was trained using the high-confidence matches from the Stage-1 match set.

Preprocessing

The following preparation was done in order to proceed with this second-level of household matching:

  1. We go back to our original set of 6962 respondent PES households.
  2. We add a new variable/column to our PES dataset, where we will indicate if this household found a match during Stage-1 matching, and we will also add the confidence score column.
  3. We apply step #2 to the Census dataset as well. (add note)

What we have now is our two separate PES and Census datasets, but with special columns that indicate if they specific household record has previously found a match during L1-matching.

Matching Algorithm

1. Merging

The merging is simpler but more expansive than the Stage-1 merging. For Stage-2, we simply merge every unmatched PES household in a given ED, with every other unmatched Census household in the ED. This means that we will end up with a dataset with of N X M total rows, where N is the number of unmatched PES households in the ED, and M is the number of unmatched census households in the ED.

Another way of describing the above merge is to say that we are doing a many-to-many merge of the unmatched PES and Census datasets, blocking by ED. The table below illustrates a simplified picture of this type of merge.

2. Scoring pairs

The scoring of the PES-Census pairs was very similar to the the Stage-1 scoring. The names of the first five household members were compared in each PES-Census pair, and a similarity score was obtained using the Jaro-Winkler method.

Each of these scores was saved in its own variable/column.

3. Selecting matches

In Stage-1 matching, before we even looked at the member names, we already knew that we were comparing households in the same exact physical/geographic location. Any discovered similarity between member names could immediately determine the pair of households to be a true match.

Now, we have more than just 1 pair of PES-Census households per geographic poin, given that we have matched an compared each remaining PES household with every remaining Census household in the ED. How do we select THE true matching pair among all these combinations?

Intuitively, we could simply say that we simply select the pair with the highest sum total of JW similarity scores. Easy, right? Well, there are a few problems with this approach.

  1. This method would ALWAYS find a match for each PES household, even if there exists no true match in the Census dataset.
  2. The score sum total is not a good metric to start with, because the sum is cumulative and is influenced by the number of persons being compared.
    • Imagine a 2-member PES household. When member names are compared with a random 10-member census household, the sum of the name similarity scores is likely to be higher than that obtained when comparing it with the true Census match.

Problem #1 can be partially solved by establishing some sort of minimum threshold to qualify a match, but again this does not solve problem #2. So to avoid getting into developing a complex algorithm to tackle every issue and edge case, we might rather decide to use a machine-learning algorithm.

For Stage-2 matching, I selected a Random Forest machine-learning algorithm for selecting matches. I will not get into the details of model, selection and evaluation, but I will mention that a logistic regression model was also evaluated but proved to be less accurate than the random forest model.

A simplified summary of the procedure:

  1. The random forest model was built using the following formula: y = score_p1 + score_p2 + score_p3 + score_p4 + score_p5 where y is the match status (Match/NonMatch) and score_pn are the JW scores obtained during scoring.

  2. The model was trained using the high-confidence matches from Stage-1.

    • Side-note: this is why we appended the ‘match’ and ‘confidence’ columns to our PES and Census datasets during the Merging step.

  3. Whenever the model predicted Y to be a “Match”, this household pair was selected as a true match.

Results

A total of 932 household-pairs were matched during Stage-2 matching. This accounts for 24.9% of the set of 3739 PES unmatched respondent households.

The table below shows a random sample of 6 matches resulting from the Random Forest approach. We can see that most seem to be true matches, with perhaps 1 being questionable (Luis Garcia). We also note, by looking at the household IDs, that the households seem to be physically close to each other. This demonstrates that this approach finds matching households even when there was a discrepancy in their building/dwelling/household numbers within the ED.

The total matches now stood at 4155 PES households, which represents 60% of respondent PES households.

Household Matching - Stage 3

Stage-3 matching of was very similar to Stage-2 matching. The same procedures were used merging, scoring, and selecting matches. This means that the Random Forest machine-learning model was used again.

The main difference in Stage-3 matching is that matching now happened up to the Supervisory Area (SA) level. (NOTE). This means that every remaining unmatched households within a PES ED were compared with all Census households in the corresponding Supervisory Area. We can see now that this would result in a very large number of comparisons, and a large merged dataset.

The random forest model was used in order to select the final matches.

Results

A total of 243 matches were found at Stage-3 matching. However, it is worth noting that at this stage, matching has become somewhat inaccurate and prone to error. This is due to the fact that every household is being compared to so many other households, that false positive matches (based on member names) are bound to exist.

To mitigate this, a manual verification and unmatching mechanism was created within the PES Web Dashboard that was created for PES operations. After the manual verification step, around 211 matches remained.

Individual Matching

After household-level matches, we now turn our attentions to the individuals within. The goal is to compare the PES non-movers and out-movers with the Census listed members in order to find individual-level matches and non-matches.

For Individual-level matching, the following criteria was used (adapted from Mr. Levinson’s guidelines in the Matching Manual):

  1. The non-movers and out-movers from the PES household are compared with the members listed in the Census household.
  2. The full names must match, except for spelling variations that do not change pronunciation.
  3. The ages must be within a close range, which varies according to age-group.
    • Mr Levinson suggested the tolerances in the table below.
  4. The following variables should also match:
    • Sex
    • Country of Birth
    • Marital Status

Preprocessing

The main pre-processing for Individual matching was as follows:

  1. From the PES and Census individual-level datasets, identify all records that belong to the set of matched households.
  2. In the PES individual-level dataset, ensure only non-movers and all out-movers are included.
  3. In-movers (those who joined the household after Census enumeration), and born-afters (babies born after Census enumeration) were identified and given their corresponding matching status at once.

Matching Algorithm

The matching algorithm at the individual level is somewhat simpler than household-level. However, it is also in parts more complex.

1. Merging

The first thing we need to know is that we need to compare every eligible PES individual with every listed Census individual. Therefore our merge should achieve this. This is illustrated in the simplified example table below.

In the example above, we see that four PES household members are compared each with the three Census household members. The subsequent steps will compare records and establishes matches.

2. Scoring Pairs

The scoring of individual-pairs was implemented based on the individual-matching guidelines outlined in the previous sections. However, a few additional enhancements were made in an attempt to improve the quality of matching. For example, weights were given to each comparison variable, given that some were more important than others.

The scoring algorithm is summarized as follows:

  1. The full names were compared using the Jaro-Winkler algorithm that was used and explained in the household-matching section.
  2. The ages were compared using a variation of the tolerances ranges suggested by Mr. Levinson.
  3. Sex, country of birth, and marital status were compared strictly: they either matched exactly or not at all.

Recognizing that not all variables should hold the same weight when scoring individual-pairs, weights were developed for each variable. Additionally, weights were developed for both a comparison success or failure. To understand this, consider the variable sex:

  • Any two records have a probability of ~50% to match successfuly on sex, even if they do not belong to the same individual (because we only have two main categories for sex). Therefore, the success weight for the variable sex should be relatively small.

  • However, if two records do not match on sex, then it is very likely that they records do not correspond to the same individual. Therefore, the failure weight for sex should be relatively large.

The determination of the success and failure weights for each variable was a mostly arbitrary decision using own judgement. (NOTE) These can certainly be refined and adjusted in the future. The weights were as follows:

  1. Name
    • success weight: 3
    • failure weight: -1
  2. Sex
    • success weight: 0.5
    • failure weight: -2
  3. Age
    • success weight: 2.5
    • failure weight: -1
  4. Marital status
    • success weight: 1
    • failure weight: 1
  5. Country of birth
    • success weight: 2
    • failure weight: -0.5

3. Selecting Matches

Matches were determined based on the scoring and weighting explained in the previous section.

  1. First, a total score was developed for each individual-pair comparison, taking into consideration the positive and negative weights for each variable.
  2. A total score threshold was established as the cut-off point for accepting or rejecting matches.
    • After experimenting a while, the threshold was established at +4.5
  3. Individual-pairs that passed the threshold were likely to belong to the same individual, and were selected as a MATCH.
  4. In the case where multiple individual-pairs passed the threshold, the pair with the highest score was selected as the MATCH.
  5. At the end of the day, each PES individual had one of the following match statuses:
    • MATCH
    • NON-MATCH
    • IN-MOVER
    • BORN-AFTER
  6. Similarly, each Census individual had one of the following match statuses:
    • MATCH
    • NON-MATCH
